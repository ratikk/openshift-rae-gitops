HHS / TIERS · IDEV Release Procedure

Releasing a build — the tag is the trigger

Jenkins builds and pushes to JFrog. An ops engineer bumps one line in git. Argo CD deploys. Nothing deploys unless a tag changes.

REPO ratikk/openshift-rae-gitopsENVS idev1 / idev2 / idev3SYNC automatic (~3 min)

01 · The flow

Three actors, two handoffs

02 · To deploy a new build

One line, one commit

1
Confirm Jenkins pushed the image to JFrog. The build you want must exist as a tag in JFrog (e.g. 122.1.0). Jenkins' job ends here — it does not touch git.
2
Edit the one tag line for the target environment. Open the per-env values file and change image.tag:
deployment/hello-openshift/values-idev1.yaml
image:
  repository: <jfrog-host>/tiers-docker/hello-openshift
  tag: "122.0.0"   # ← change to the new tag, e.g. "122.1.0"
3
Commit and push. 
# clean, auditable one-line diff
git commit -am "release: idev1 → 122.1.0"
git push
4
Argo CD deploys automatically (~3 min). It detects the diff in values-idev1.yaml, re-renders the chart with the new tag, and rolls out. Only idev1 changes — idev2 and idev3 are untouched.
Each env is independent idev1, idev2, idev3 each read their own values-idevN.yaml. Bumping one env's tag never affects the others — that is the parallel-release model. One env can run a build under test while another runs the next dev build.

03 · Why "only on tag change"

Argo deploys on diffs, not on schedules

Argo CD continuously compares desired state (git) against live state (cluster). It acts only when they differ. The single thing that changes desired state for a release is the engineer editing image.tag. So:

ActionArgo behavior
Jenkins pushes a new image to JFrog (no git change)Nothing — git unchanged, no diff
Engineer bumps image.tag + commitsDeploys — one-line diff detected
Re-run a build with the same tagNothing — tag value identical
Someone hand-edits the live DeploymentReverted — selfHeal restores git's tag

You do not configure a "trigger." The git commit is the trigger; the absence of a commit is the absence of a deploy.

04 · JFrog pull credentials (via Vault + ESO)

The private-registry secret never touches git

Pulling from private JFrog needs a dockerconfigjson pull Secret in each namespace. It is not committed — it is fetched from Vault by External Secrets, reusing the stack already in the cluster.

One-time: store JFrog creds in Vault

oc -n vault-system exec vault-0 -- vault kv put secret/jfrog \
  username='<jfrog-user>' \
  password='<jfrog-token>' \
  host='<jfrog-host>'

Per namespace: the ExternalSecret materializes the pull Secret

externalsecret-jfrog.yaml (one per idev namespace) reads secret/data/jfrog via the lab-vault ClusterSecretStore and renders a Secret named jfrog-pull. The chart's imagePullSecret: jfrog-pull then uses it. Validate:

# the pull secret was materialized by ESO
oc -n idev1 get secret jfrog-pull -o jsonpath='{.type}{"\n"}'
# → kubernetes.io/dockerconfigjson

# the ExternalSecret is healthy
oc -n idev1 get externalsecret jfrog-pull
Why this way The image-pull credential is a secret like any other — so it flows Vault → ESO → namespace Secret, exactly like application secrets. Nothing sensitive lands in the GitOps repo; only a reference to the Vault path does.

05 · Verify a release landed

# the running image matches the tag you committed
oc -n idev1 get deploy hello-openshift \
  -o jsonpath='{.spec.template.spec.containers[0].image}{"\n"}'

# Argo shows the app Synced + Healthy on the new revision
oc -n openshift-gitops get application idev1-hello-openshift

# the app responds with its release string
curl -sk "https://$(oc -n idev1 get route -o jsonpath='{.items[0].spec.host}')"
If it didn't deploy Check, in order: (1) did the commit actually change image.tag? git show; (2) is the Argo app Synced? if OutOfSync, the auto-sync may be paused; (3) ImagePullBackOff? the jfrog-pull secret is missing or the tag doesn't exist in JFrog.
